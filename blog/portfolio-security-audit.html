La duda que no me dejaba tranquilo

Cuando decidí crear mi portfolio web, había algo que me preocupaba más que el diseño o las funcionalidades: la seguridad.

No soy desarrollador profesional, aunque tengo conocimientos técnicos básicos. Pero este proyecto tenía algo diferente — lo construí prácticamente entero usando IA, concretamente con Claude Code.

Y ahí empezó mi duda constante:

"¿Y si la IA comete alguna mala práctica de seguridad sin que yo me dé cuenta?"

Aunque fui muy insistente con buenas prácticas, validaciones y protección de datos, seguía teniendo esa espina. Al final, cuando no eres especialista en ciberseguridad, siempre queda la incertidumbre.

Así que decidí hacer lo lógico: pedí una auditoría completa de seguridad del proyecto y del repositorio en GitHub mientras estaba en funcionamiento.

Buenas noticias: no se encontró nada peligroso. El repositorio sigue buenas prácticas de seguridad en todas las áreas revisadas.

Puntos especialmente positivos

Más allá de "no hay problemas", hay prácticas que el audit destacó como especialmente correctas:

  • El .env real nunca se sube a Git — está correctamente excluido vía .gitignore
  • Las API keys se cargan mediante process.env, no están hardcodeadas en el código
  • Los emails se hashean con SHA-256 antes de enviarse a analytics
  • Las consultas a Supabase usan ORM — no SQL directo, lo que elimina el riesgo de inyección
  • El formulario de contacto valida inputs antes de procesarlos

Conclusión directa: el repositorio y el dispositivo no están en riesgo por este proyecto. Las prácticas de seguridad implementadas son correctas.

Mejoras opcionales (no urgentes)

El audit también identificó tres ajustes recomendados para reforzar aún más la seguridad a futuro. Ninguno es crítico hoy, pero son buenas prácticas para tener en cuenta:

⚠
Rate limiting en /api/subscribe y /api/contact para evitar spam o abuso automatizado de los formularios.
⚠
Tokens CSRF en los formularios para protección adicional contra solicitudes falsificadas entre sitios.
⚠
Ejecutar npm audit periódicamente para detectar nuevas vulnerabilidades en dependencias a medida que se publican.

Lo que aprendí de todo esto

La conclusión más importante no es técnica — es de mentalidad:

Usar IA para desarrollar no significa descuidar la seguridad. Pero sí implica algo clave: la IA acelera el desarrollo. La responsabilidad sigue siendo humana.

Este proceso me enseñó que, incluso sin ser desarrollador profesional, puedes crear proyectos sólidos si te apoyas en buenas herramientas, sigues buenas prácticas, y verificas con auditorías independientes.

¿Qué garantiza realmente la seguridad en un proyecto con IA?

Reflexionando sobre el proceso, identifiqué tres factores que marcaron la diferencia:

1. Ser muy explícito con las instrucciones. Fui insistente con buenas prácticas desde el primer prompt: variables de entorno, validación de inputs, no hardcodear credenciales. La IA cumplió porque lo pedí con claridad.

2. Entender el código que genera. No necesitas ser experto, pero sí entender a grandes rasgos qué hace cada parte. Si algo no te cuadra, pregunta. Esa revisión humana es lo que convierte un borrador de IA en código producción-ready.

3. Auditar independientemente. La validación externa — aunque sea interna al proyecto — es lo que cierra el ciclo. No asumas que todo está bien; compruébalo.

La seguridad no es un estado final. Es un proceso continuo de revisión, auditoría y mejora.

Si también estás construyendo con IA

Si alguien está pensando en crear proyectos con IA pero le preocupa la seguridad, mi experiencia es clara:

  • Es una preocupación válida — y lo será cada vez más a medida que los proyectos crezcan
  • Pero se puede gestionar con las prácticas adecuadas desde el inicio
  • Y auditar te da tranquilidad real, no una sensación falsa de seguridad

El stack que usé — Next.js + Supabase + Vercel + Claude Code — tiene buenas defensas por defecto si lo configuras bien. La clave es no delegar esa configuración por completo a la IA sin revisarla.
